# מודל חיוב - אפיון (בסגנון iCount)

מוטי בחר במודל חיוב כמו iCount (מסלול Advanced). תלוי ב-API של iCount לחיוב חוזר
(הו"ק) - לא ניתן לבנייה מלאה עד שמתקבלים פרטי ה-API. זה ה-blueprint.

## העקרונות (כפי שמוטי הגדיר)
1. **חיוב חודשי בדיעבד:** בתחילת כל חודש קלנדרי מחויב השימוש בפועל של **החודש הקודם**.
2. **פרו-רטה למודולים:** מודול שהופעל באמצע החודש מחויב יחסית למספר הימים שהיה פעיל.
3. **חיוב מרוכז ב-1 לחודש:** חיוב אחד שכולל את **המסלול הבסיסי** + החלק היחסי של כל מודול שהופעל.
4. **חיוב חד-פעמי = מיידי:** עלות הקמה / תוסף חד-פעמי (למשל ₪149 תגי מעקב) מחויבת ומופקת לה חשבונית **מיד עם ההשלמה**, בלי להמתין למחזור החודשי.

## איך זה ממופה אצלנו
- **מסלול בסיס:** החבילה החודשית (₪59/79/99/149).
- **מודולים (פרו-רטה):** CRM ₪49, ביקורות ₪14/חודש, מייל עסקי, וואטסאפ, דיוור - כל אחד מחושב לפי ימי-פעילות בחודש.
- **חד-פעמי (מיידי):** תגי מעקב ₪149, רכישת דומיין, חבילות קרדיט AI.

## רכיבים לבנייה (כשiCount מוכן)
1. **טבלת `subscription_items`** - לכל עסק: רשימת המודולים הפעילים + תאריך הפעלה/כיבוי (ל-pro-rata).
2. **מנוע חישוב חודשי** - cron ב-1 לחודש: לכל עסק מסכם בסיס + סך פרו-רטה של מודולים → סכום חודשי.
3. **אינטגרציית iCount (הו"ק)** - יצירת חיוב חוזר + הפקת חשבונית מרוכזת. דורש iCount API.
4. **חד-פעמי מיידי** - על השלמת רכישה: חיוב + חשבונית מיד (לא דרך המחזור החודשי).
5. **תצוגה בדשבורד** - ב"התוכנית שלי": פירוט החיוב הצפוי (בסיס + מודולים), חשבוניות (כבר יש `icount-invoices`).

## חישוב פרו-רטה (נוסחה)
`חיוב מודול = מחיר_חודשי × (ימים_פעיל_בחודש / ימים_בחודש)`

## תלות

עד אז: בונים את הטבלאות + מנוע החישוב + התצוגה, ומשאירים את שלב החיוב בפועל מאחורי דגל.
