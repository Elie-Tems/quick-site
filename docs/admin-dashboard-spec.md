# אפיון: דשבורד העל (אדמין) כמרכז שליטה מבוסס-ניהול לקוחות

מסמך אפיון לארגון מחדש של **דשבורד העל של סיאנגו** (`/manage-x7k9`) - המערכת הפנימית שבה מוטי והצוות מנהלים את כל הסוחרים בפלטפורמה. המטרה: להפוך אותו ממערכת עשירה-אך-מפוזרת (8 קבוצות, ~30 מסכים) ל-**מרכז שליטה שבו הסוחר הוא הלקוח** (Merchant ניהול לקוחות), עם אזור **הכנסות ודוחות** מאוחד - בלי להוריד אף יכולת קיימת.

מבוסס על מיפוי מלא של האדמין הקיים.

---

## 1. החזון

באדמין, **הסוחרים הם הלקוחות שלנו.** היום המידע על כל סוחר מפוזר: רשימת לקוחות במסך אחד, רשימת חנויות באחר, חיובים בשלישי, רדומים ברביעי, נטישה בחמישי. אין מקום אחד שמראה "מי הסוחר הזה, כמה הוא שווה לנו, מה מצב הבריאות שלו, ומה לעשות איתו".

המעבר: הדשבורד נבנה מחדש סביב **ניהול לקוחות של סוחרים** - כרטיס-על 360 לכל סוחר (זהות + חנות + הכנסה חודשית קבועה + חיובים + בריאות + סיכון נטישה + ציר זמן + פעולות) - ואזור **הכנסות ודוחות** מאוחד עם מקור-אמת אחד.

**3 עקרונות:**
1. **שום יכולת לא נמחקת.** כל ~30 המסכים ממשיכים לעבוד - רק מתארגנים ומתחברים. טבלת מיפוי (סעיף 7).
2. **מקור-אמת אחד.** היום הכנסה חודשית קבועה מחושב ב-3 מקומות שונים עם לוגיקה שונה. מאחדים.
3. **סוגרים פערים קיימים.** קוד מת (`AdminUsersList` בנוי ולא מחובר), הגדרות ריקות, אין ניהול לקוחות-סוחר מאוחד, ייצוא מפוזר.

---

## 2. הבעיה היום

מהמיפוי המלא עלו הפערים הבאים:
- **אין ניהול לקוחות-סוחר מאוחד.** `AdminCustomers` (רשימת אנשים), `AdminBusinesses` (רשימת חנויות), `AdminDormant` (רדומים), וחיובים פר-לקוח - כולם חתיכות נפרדות שלא מחוברות לכרטיס אחד.
- **קוד מת + כפילות.** `AdminUsersList` בנוי במלואו אבל **לא מחובר** לשום מקום; חופף ל-`AdminCustomers`. שתי תצוגות "אנשים" מקבילות.
- **הכנסה חודשית קבועה ב-3 מקומות** (Command Center, Overview, מסך הכנסה חודשית קבועה) עם 3 חישובים שונים ו-fallback קשיח ₪69. אין מקור-אמת.
- **אין אזור דוחות מאוחד.** ייצוא CSV מפוזר פר-מסך; מסכי האנליטיקס קריאה-בלבד בלי מרכז ייצוא/דוחות.
- **הגדרות = placeholder ריק.** אין ניהול הפעלה/כיבוי מודולים, אין ניהול הרשאות/אדמינים, אין אכיפת אימות דו-שלבי.
- **3 משפכים נפרדים** (הרשמה / קונים / UTM) בלי קישור ביניהם.
- **תקרות סקייל** (300 חנויות, 100 הזמנות, בלי pagination בצד-שרת).

---

## 3. המבנה החדש - 7 אזורים

```
1. מרכז שליטה           - דופק, KPI, התראות חכמות, פעילות חיה
2. ניהול הסוחרים          ⭐ הליבה
     • כל הסוחרים (directory: חיפוש, פילטרים, בריאות, שלב מחזור-חיים)
     • כרטיס סוחר 360 (זהות + חנות + הכנסה חודשית קבועה + חיובים + בריאות + סיכון נטישה + ציר זמן + הערות/פעולות)
     • בסיכון / רדומים (churn + dormant מאוחדים)
     • הזמנות סוחרים
3. הכנסות ודוחות         ⭐ מאוחד
     • הכנסה חודשית / שנתית (מקור-אמת אחד)
     • מסלול הרשמה · נטישה · שימור לאורך זמן retention
     • הכנסות סיאנגו מול מחזור מכירות סוחרים
     • תמונת שוק · Top performers · קטגוריות · צפיות
     • דוחות וייצוא (מרכז ייצוא אחד)
4. תשלומים               - תשלומים · ביטולים · שגיאות תשלום · החזרים
5. תמחור ושותפים         - דומיינים · מייל · קופוני מנוי · רווחי שותפים · שיווק (רכישת לקוחות של סיאנגו)
6. תקשורת ודיוור         - וואטסאפ · הבוט שלנו · רשימת הסרות · לוג מיילים
7. מערכת                 ⭐ (נבנה מה-placeholder) - הגדרות · הפעלה/כיבוי מודולים · הרשאות ואדמינים · אימות דו-שלבי
```

**השינויים המרכזיים מול היום:**
- **"ניהול הסוחרים" נולד** - מאחד את AdminCustomers + AdminBusinesses + הקוד-המת AdminUsersList + AdminDormant + החיובים-פר-לקוח לכרטיס-על אחד עם ציון בריאות ושלב מחזור-חיים.
- **"הכנסות ודוחות" מאוחד** - מאחד את קבוצות "הכנסות" + "תובנות ושוק" הנפרדות; מקור-אמת אחד ל-הכנסה חודשית קבועה; מרכז ייצוא.
- **"מערכת" נבנה** מה-placeholder הריק - הפעלה/כיבוי מודולים, הרשאות, אימות דו-שלבי.

---

## 4. ניהול הסוחרים (הליבה)

### 4.1 Directory - כל הסוחרים
טבלה/כרטיסים אחודים של כל סוחר, עם עמודות שהיום מפוזרות: שם, מייל, טלפון, חנות, קטגוריה, תוכנית, פורסם, תאריך הצטרפות, **הכנסה חודשית קבועה פר-סוחר**, **שלב מחזור-חיים** (חדש / פעיל / בסיכון / רדום / נוטש), **ציון בריאות**. חיפוש + פילטרים + pagination בצד-שרת (סוגר את תקרת ה-300).

### 4.2 כרטיס סוחר 360 - ⭐ חדש
לחיצה על סוחר פותחת כרטיס-על אחד שמרכז את מה שהיום מפוזר ב-5 מסכים:
- **זהות + חנות** (מ-AdminCustomers/AdminBusinesses).
- **הכנסה חודשית קבועה + היסטוריית חיובים** (BillingChargesPanel הקיים) + **החזר**.
- **ציון בריאות ושלב מחזור-חיים** (פעילות, הזמנות אחרונות, סטטוס תשלום).
- **סיכון נטישה** (מ-churn/dormant).
- **ציר זמן פעילות** (הזמנות, פרסום, תשלומים, פניות).
- **פעולות אדמין**: מייל, צפייה בחנות, איפוס onboarding, השהיה/מחיקה, החזר, **הערות פנימיות** (חדש - ניהול לקוחות אמיתי).

### 4.3 בסיכון / רדומים
מאחד `AdminChurnRate` + `AdminDormant` + לקוחות "בסיכון" לרשימת פעולה אחת (מי לתפוס לפני שנוטש).

### 4.4 הזמנות סוחרים
`AdminOrdersList` הקיים (כל ההזמנות בפלטפורמה) - עובר לכאן, עם pagination.

## 5. הכנסות ודוחות (מאוחד)

מאחד את קבוצות "הכנסות" ו"תובנות ושוק" הנפרדות למקום אחד:

| תת-מסך | מקור | סטטוס |
|---|---|---|
| **הכנסה חודשית / שנתית** | `Adminהכנסה חודשית קבועה` - **מקור-אמת אחד** (מבטל 3 החישובים) | קיים, מאוחד |
| **מסלול הרשמה · נטישה · שימור לאורך זמן** | Adminמסלול הרשמה / ChurnRate / שימור לאורך זמןRetention | קיים |
| **הכנסות סיאנגו מול מחזור מכירות סוחרים** | AdminSiangoRevenue מול Adminתמונת שוק/Payments | קיים, מוצג צד-לצד |
| **Top performers · קטגוריות · צפיות** | AdminTopPerformers / CategoryMap / Analytics | קיים |
| **דוחות וייצוא** | מרכז ייצוא אחד (CSV/PDF) לכל הדוחות | חדש (מחליף ייצוא מפוזר) |

**מקור-אמת ל-הכנסה חודשית קבועה** הוא התיקון החשוב: היום הכנסה חודשית קבועה מחושב ב-3 מקומות עם תוצאות שונות. מאחדים ל-hook אחד (`useהכנסה חודשית קבועה`) שכולם צורכים.

---

## 6. מערכת (נבנה מה-placeholder)

האזור "הגדרות" הריק הופך ל-**מערכת** אמיתי:
- **Feature-flags** - הפעלה/כיבוי של מודולים (whatsapp, email, crm...) שכבר מוזכרים בקופונים.
- **הרשאות ואדמינים** - ניהול תפקידי `admin` (RPC `has_role` קיים), הוספה/הסרה.
- **אימות דו-שלבי** - אכיפת הרשמה ל-2FA (היום נאכף רק אם כבר נרשם).
- **קונפיג מערכת** - ערכים גלובליים במקום fallback-ים קשיחים בקוד.

## 7. טבלת מיפוי - לאן עובר כל מסך אדמין קיים

| מסך אדמין קיים | היום (קבוצה) | מיקום חדש |
|---|---|---|
| סקירה כללית / Command Center | ראשי | **מרכז שליטה** |
| פעילות חיה | ראשי | **מרכז שליטה ← פעילות** |
| לקוחות (AdminCustomers) | לקוחות וחנויות | **ניהול הסוחרים ← directory + כרטיס 360** |
| חנויות (AdminBusinesses) | לקוחות וחנויות | **ניהול הסוחרים** (מתמזג לכרטיס 360) |
| (AdminUsersList - קוד מת) | לא מחובר | **ניהול הסוחרים** (מתמזג, מפסיק כפילות) |
| רדומים (AdminDormant) | תובנות ושוק | **ניהול הסוחרים ← בסיכון/רדומים** |
| הזמנות (AdminOrdersList) | הזמנות ותשלומים | **ניהול הסוחרים ← הזמנות** |
| תשלומים · ביטולים · שגיאות תשלום | הזמנות ותשלומים | **תשלומים** |
| הכנסה חודשית קבועה/הכנסה שנתית · מסלול הרשמה · נטישה · שימור לאורך זמן | הכנסות | **הכנסות ודוחות** (הכנסה חודשית קבועה = מקור-אמת אחד) |
| צפיות · תמונת שוק · Top · קטגוריות | תובנות ושוק | **הכנסות ודוחות** |
| הכנסות סיאנגו (SiangoRevenue) | בתוך תשלומים | **הכנסות ודוחות** (צד-לצד עם מחזור מכירות) |
| דומיינים · מייל · קופונים · שותפים · שיווק | תמחור ושותפים | **תמחור ושותפים** (נשאר) |
| וואטסאפ · בוט · הסרות · לוג מיילים | תקשורת ודיוור | **תקשורת ודיוור** (נשאר) |
| הפניות (Referrals) | הזמנות ותשלומים | **תמחור ושותפים ← הפניות** |
| הגדרות (placeholder) | הגדרות | **מערכת** (נבנה: flags/הרשאות/אימות דו-שלבי) |

**כל המסכים ממופים. אף יכולת לא נמחקת - קוד מת מחובר, כפילויות מתמזגות, מדדים מתאחדים.**

## 8. תוכנית מדורגת

- **שלב 0 - שלד ניווט.** מבנה 7-האזורים החדש + מיפוי כל view. סיכון נמוך.
- **שלב 1 - ניהול הסוחרים.** מאחדים לקוחות+חנויות+UsersList+רדומים לכרטיס 360; מוסיפים ציון בריאות + שלב מחזור-חיים + הערות; pagination בצד-שרת.
- **שלב 2 - הכנסות ודוחות.** מאחדים את שתי קבוצות האנליטיקס; מקור-אמת אחד ל-הכנסה חודשית קבועה; מרכז ייצוא.
- **שלב 3 - מערכת.** בונים הפעלה/כיבוי מודולים + הרשאות + אימות דו-שלבי.
- **שלב 4 - ליטוש.** מבטלים fallback-ים קשיחים; מחברים קופונים ל-checkout (שכבר סומן "next step").

---

## 9. מוקאפים לאישור

מוצגים בצ'אט:
1. שלד האדמין + תפריט 7-האזורים + מרכז שליטה (דופק + התראות).
2. ניהול הסוחרים - directory + כרטיס סוחר 360 (בריאות · הכנסה חודשית קבועה · סיכון נטישה).
3. הכנסות ודוחות - סקירה מאוחדת (הכנסה חודשית קבועה · נטישה · סיאנגו מול מחזור מכירות).

לאחר אישור - בונים לפי התוכנית המדורגת.
