## ⛔ כלל הזהב: קוד שלא נפרס = שבור בפרודקשן

**כל edge function שיצרת או שינית - אתה חייב לפרוס אותה מיד אחרי.**
פונקציה שקיימת בקוד אבל לא פרוסה ל-Supabase מחזירה 404 והפיצ'ר שבור אצל לקוחות,
בלי שום שגיאה בקוד. (זה בדיוק מה שקרה עם `orders-create` - נבנתה, חוברה ל-frontend,
אבל לא נפרסה → כל ההזמנות נכשלו בשקט.)

אחרי שינוי פונקציה, תמיד:
```bash
npx supabase@latest functions deploy <function-name> --project-ref ytqgeoviokgxxwalieev
```
(דורש `SUPABASE_ACCESS_TOKEN` בסביבה.)

**אם הוספת פונקציה חדשה שנקראת מחנות ציבורית** (storefront, לקוח לא מחובר) - הוסף לה
`verify_jwt = false` ב-`supabase/config.toml`, אחרת היא תדחה את הלקוחות.

## פריסה (Deploy) - איך הכול עולה לאוויר

| מה | איך עולה לפרודקשן |
|---|---|
| **Frontend** (קוד React/Vite) | אוטומטי - כל `git push` ל-`main` מפעיל GitHub Action שבונה ופורס ל-Cloudflare Pages. אין צורך לפרוס ידנית. |
| **Edge functions** | **ידני** - `supabase functions deploy <name>` (ראה כלל הזהב). |
| **שינויי DB / טבלאות** | רק דרך migration ב-`supabase/migrations/` - לא לשנות סכמה ידנית בדשבורד. |
| **Secrets / API keys** | רק בדשבורד של Supabase (Edge Functions → Secrets). **לעולם לא בקוד, לא ב-git, לא בצ'אט.** |

## עבודה משותפת על `main` (מוטי + דניאל)

שנינו דוחפים לאותו ענף `main`. כדי לא לדרוס אחד את השני:
1. **תמיד `git pull origin main` לפני שמתחילים לעבוד.**
2. **לפני שמוחקים או מחליפים קובץ/רכיב קיים - לקרוא מה יש בו.** אם בנו שם משהו
   (הדרכה, לוגיקה, טקסט) - לא להפיל אותו בטעות. (כך אבדו ההדרכה וספקי ה"בקרוב"
   כש-`StepPayments` הוחלף ב-`StepFinish`.)
3. שינויים קטנים וממוקדים + `git push` מהר, כדי לצמצם התנגשויות.
4. הודעות commit ברורות בעברית/אנגלית שמסבירות **למה**, לא רק מה.
